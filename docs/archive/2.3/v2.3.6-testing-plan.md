## Test Suite 2: Vulnerability Investigation Workflow

### Test 2.1: Automatic Investigation Trigger

**Setup:**
1. Ensure BurpSuite is activated (from Test Suite 1)
2. Create or select an active operation
3. Ensure operation has at least one target

**Trigger Investigation:**
```bash
# Option A: Via BBOT scan (creates vulnerabilities automatically)
curl -X POST http://localhost:3001/api/v1/surface-assessment/scan \
  -H "Content-Type: application/json" \
  -H "Cookie: connect.sid=YOUR_SESSION" \
  -d '{
    "operationId": "YOUR_OPERATION_ID",
    "targets": ["example.com"],
    "scanType": "bbot",
    "options": { "modules": ["httpx", "nuclei"] }
  }'

# Option B: Create a vulnerability manually
curl -X POST http://localhost:3001/api/v1/vulnerabilities \
  -H "Content-Type: application/json" \
  -H "Cookie: connect.sid=YOUR_SESSION" \
  -d '{
    "title": "Test SQL Injection",
    "description": "Testing vulnerability investigation workflow",
    "severity": "high",
    "cveId": "CVE-2023-12345",
    "operationId": "YOUR_OPERATION_ID",
    "targetId": "YOUR_TARGET_ID",
    "investigationStatus": "pending"
  }'
```

**Expected Behavior:**
1. Vulnerability created with `investigationStatus: "pending"`
2. Workflow automatically triggered (check server logs)
3. Status changes to `investigating`
4. R&D Team Agent checks for existing Nuclei template
5. If no template: R&D Team Agent researches CVE via Tavily
6. BurpSuite Orchestrator Agent investigates the finding
7. Status updates to `validated`, `false_positive`, or `inconclusive`

**Server Logs to Monitor:**
```bash

# Watch R&D Team Agent logs

# Watch BurpSuite Orchestrator logs
```

**Expected Log Output:**

### Test 6.1: Database Migration Verification

**Query to verify table structure:**
```sql
-- Verify burpSetup table exists
SELECT table_name, column_name, data_type 
FROM information_schema.columns 
WHERE table_name = 'burp_setup' 
ORDER BY ordinal_position;

-- Verify investigation columns on vulnerabilities table
SELECT column_name, data_type, column_default
FROM information_schema.columns 
WHERE table_name = 'vulnerabilities' 
AND column_name LIKE 'investigation%';

-- Verify nucleiTemplates table
SELECT table_name, column_name, data_type 
FROM information_schema.columns 
WHERE table_name = 'nuclei_templates' 
ORDER BY ordinal_position;

-- Verify foreign key constraints
SELECT
    tc.constraint_name,
    tc.table_name,
    kcu.column_name,
    ccu.table_name AS foreign_table_name,
    ccu.column_name AS foreign_column_name
FROM information_schema.table_constraints AS tc
JOIN information_schema.key_column_usage AS kcu
    ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage AS ccu
    ON ccu.constraint_name = tc.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
    AND (tc.table_name = 'vulnerabilities' OR tc.table_name = 'burp_setup' OR tc.table_name = 'nuclei_templates');
```

**Expected Results:**
- ✅ `burp_setup` table with all activation fields
- ✅ `vulnerabilities` table has investigation_status, investigation_assigned_at, investigation_completed_at, burp_scan_reference, nuclei_template_id, validation_evidence, investigation_agent_id columns
- ✅ `nuclei_templates` table with generated_by_ai, target_vulnerability_id fields
- ✅ All foreign keys properly defined

## Troubleshooting Guide

### Issue: Activation Fails
**Symptoms:** Status stuck in "Activating" or changes to "Error"

**Diagnostics:**
```bash
# Check activation log in database
psql -U rtpi -d rtpi -c "SELECT activation_log, error_message FROM burp_setup;"

# Check container logs
docker logs rtpi-burp-agent --tail 100

# Check if files copied to container
docker exec rtpi-burp-agent ls -lah /opt/burp-setup/

# Test MCP server manually
docker exec rtpi-burp-agent curl -v http://localhost:9876/health
```

**Common Fixes:**
- Ensure JAR file is valid (check magic bytes)
- Verify license is not expired
- Restart rtpi-burp-agent container
- Check Docker volume mounts

### Issue: Investigation Not Triggering
**Symptoms:** Vulnerability created but stays in "pending" status

**Diagnostics:**
```bash
# Check server logs for workflow events
docker compose logs server | grep "Investigation Workflow"

# Check vulnerability severity
psql -U rtpi -d rtpi -c "SELECT id, title, severity, investigation_status FROM vulnerabilities WHERE id = 'YOUR_VULN_ID';"

# Manually trigger investigation
curl -X POST http://localhost:3001/api/v1/vulnerability-investigation/YOUR_VULN_ID/investigate \
  -H "Cookie: connect.sid=YOUR_SESSION"
```

**Common Fixes:**
- Verify severity is medium/high/critical (not low/informational)
- Check if already investigated (non-pending status)
- Ensure operation is active
- Restart server to reset event listeners

### Issue: R&D Team Agent Fails
**Symptoms:** Template generation errors or empty research results

**Diagnostics:**
```bash
# Check Tavily API key
docker compose exec server env | grep TAVILY

# Test Tavily directly
curl -X POST https://api.tavily.com/search \
  -H "Content-Type: application/json" \
  -d '{"api_key": "YOUR_KEY", "query": "CVE-2024-1234", "max_results": 5}'

# Check Ollama availability
curl http://localhost:11434/api/tags
```

**Common Fixes:**
- Set TAVILY_API_KEY in .env file
- Verify Ollama is running and models are loaded
- Check agent initialization in logs
