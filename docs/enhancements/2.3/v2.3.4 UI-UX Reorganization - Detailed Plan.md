## Step 4.5: Multi-Agent Loop Blocks (Per-Operation)

### Purpose

Currently, each agent supports only a **single loop partner** (`loopPartnerId` in `AgentConfig`), loops have **no operation context**, and loop state is stored entirely in memory (`AgentLoopService.activeLoops` Map). This step extends the loop system to support:

1. **Multiple loops per agent** — an agent can participate in several loop partnerships simultaneously
2. **Block-based flow diagram** — loops are visualized as a horizontal pipeline of connected agent nodes
3. **Automatic cross-agent display** — viewing any agent in a chain reveals the entire connected loop topology
4. **Per-operation scoping** — loop configurations are tied to a specific operation, so the same agents can have different loop arrangements across different engagements

### Example: Current Agent Configuration

Based on the existing agent setup (from the Edit Agent dialog):

| Agent | Type | Loop Partner | MCP Server | Tools | Exit Condition |
|-------|------|-------------|------------|-------|----------------|
| **Operation Lead** | Anthropic | Senior Cyber Operator | — | 0 | Exploit Successful |
| **Senior Cyber Operator** | Anthropic | *(hub — referenced by both)* | — | — | — |
| **Research Agent** | Custom | Senior Cyber Operator | Tavily Remote MCP | 3 | Exploit Successful |

This forms the loop chain:

```
Operation Lead  ↔  Senior Cyber Operator  ↔  Research Agent
```

In the new system, this is represented as a single **Loop Block** scoped to an operation:

```
┌──────────────────────────────────────────────────────────────────────────────────┐
│  Operation: Acme Corp Pentest                                         [+ Add]   │
│                                                                                  │
│  ┌──────────────┐         ┌─────────────────────┐         ┌──────────────────┐  │
│  │ Operation    │ ──5i──▶ │ Senior Cyber        │ ──5i──▶ │ Research Agent   │  │
│  │ Lead         │ ◀────── │ Operator            │ ◀────── │                  │  │
│  │ (Anthropic)  │         │ (Anthropic)         │         │ (Custom)         │  │
│  │              │         │                     │         │ MCP: Tavily      │  │
│  │ Exit: Exploit│         │    Hub Agent        │         │ Tools: 3         │  │
│  │ Successful   │         │                     │         │ Exit: Exploit    │  │
│  └──────────────┘         └─────────────────────┘         │ Successful       │  │
│                                                           └──────────────────┘  │
│  ─────────────────────────────────────────────────────────────────────────────── │
│  Block Order: 1 → 2 → 3          Status: Ready          [▶ Start Chain]        │
└──────────────────────────────────────────────────────────────────────────────────┘
```

- **5i** = 5 max iterations per loop link
- **Hub Agent** = agent participating in 2+ loop partnerships (automatically labeled)
- **Block Order** = execution sequence (drag blocks to reorder)

### Schema Changes

**File:** `shared/schema.ts` (MODIFY)

Add three new tables after the existing `agents` table definition (line ~331):

```typescript
// ── Operation-scoped loop blocks ──────────────────────────────────

export const operationLoopBlocks = pgTable("operation_loop_blocks", {
  id: uuid("id").primaryKey().defaultRandom(),
  operationId: uuid("operation_id").notNull().references(() => operations.id),
  name: text("name"),                         // Optional label, e.g. "Recon Chain"
  createdAt: timestamp("created_at").notNull().defaultNow(),
  updatedAt: timestamp("updated_at").notNull().defaultNow(),
});

export const loopBlockAgents = pgTable("loop_block_agents", {
  id: uuid("id").primaryKey().defaultRandom(),
  blockId: uuid("block_id").notNull().references(() => operationLoopBlocks.id, { onDelete: "cascade" }),
  agentId: uuid("agent_id").notNull().references(() => agents.id),
  position: integer("position").notNull(),    // Order within the block (0, 1, 2...)
});

export const loopBlockLinks = pgTable("loop_block_links", {
  id: uuid("id").primaryKey().defaultRandom(),
  blockId: uuid("block_id").notNull().references(() => operationLoopBlocks.id, { onDelete: "cascade" }),
  fromAgentId: uuid("from_agent_id").notNull().references(() => agents.id),
  toAgentId: uuid("to_agent_id").notNull().references(() => agents.id),
  maxIterations: integer("max_iterations").notNull().default(5),
  exitCondition: text("exit_condition").notNull().default("functional_poc"),
});
```

**Rationale:** Separating loop topology from the `agents.config` JSON column allows the same agent to participate in different loop arrangements across operations without config collisions. The legacy `loopPartnerId` field remains for backwards compatibility.

### Backend — Loop Block API Routes

**File:** `server/api/v1/loop-blocks.ts` (NEW)

```typescript
import { Router } from "express";
import { ensureAuthenticated, logAudit } from "../../auth/middleware";
import { db } from "../../db";
import { operationLoopBlocks, loopBlockAgents, loopBlockLinks, agents } from "@shared/schema";
import { eq, asc } from "drizzle-orm";

const router = Router();
router.use(ensureAuthenticated);

// GET /api/v1/operations/:operationId/loop-blocks
// Returns all loop blocks for an operation, with their agents and links
router.get("/operations/:operationId/loop-blocks", async (req, res) => {
  const { operationId } = req.params;
  try {
    const blocks = await db
      .select()
      .from(operationLoopBlocks)
      .where(eq(operationLoopBlocks.operationId, operationId));

    // For each block, fetch agents (ordered by position) and links
    const result = await Promise.all(blocks.map(async (block) => {
      const blockAgents = await db
        .select({ id: loopBlockAgents.id, agentId: loopBlockAgents.agentId, position: loopBlockAgents.position })
        .from(loopBlockAgents)
        .where(eq(loopBlockAgents.blockId, block.id))
        .orderBy(asc(loopBlockAgents.position));

      const links = await db
        .select()
        .from(loopBlockLinks)
        .where(eq(loopBlockLinks.blockId, block.id));

      // Resolve agent details
      const agentIds = blockAgents.map(a => a.agentId);
      const agentDetails = agentIds.length > 0
        ? await db.select().from(agents).where(inArray(agents.id, agentIds))
        : [];

      return {
        ...block,
        agents: blockAgents.map(ba => ({
          ...ba,
          agent: agentDetails.find(a => a.id === ba.agentId),
        })),
        links,
      };
    }));

    res.json({ blocks: result });
  } catch (error: any) {
    res.status(500).json({ error: "Failed to list loop blocks", details: error?.message });
  }
});

// POST /api/v1/operations/:operationId/loop-blocks
// Create a new loop block with agents and links
router.post("/operations/:operationId/loop-blocks", async (req, res) => {
  const user = req.user as any;
  const { operationId } = req.params;
  const { name, agents: blockAgentList, links } = req.body;
  // blockAgentList: [{ agentId: string, position: number }]
  // links: [{ fromAgentId: string, toAgentId: string, maxIterations: number, exitCondition: string }]

  try {
    const [block] = await db.insert(operationLoopBlocks).values({
      operationId,
      name: name || null,
    }).returning();

    if (blockAgentList?.length) {
      await db.insert(loopBlockAgents).values(
        blockAgentList.map((a: any) => ({ blockId: block.id, agentId: a.agentId, position: a.position }))
      );
    }

    if (links?.length) {
      await db.insert(loopBlockLinks).values(
        links.map((l: any) => ({
          blockId: block.id,
          fromAgentId: l.fromAgentId,
          toAgentId: l.toAgentId,
          maxIterations: l.maxIterations || 5,
          exitCondition: l.exitCondition || "functional_poc",
        }))
      );
    }

    await logAudit(user.id, "create_loop_block", "/loop-blocks", block.id, true, req);
    res.status(201).json({ block });
  } catch (error: any) {
    res.status(500).json({ error: "Failed to create loop block", details: error?.message });
  }
});

// PUT /api/v1/loop-blocks/:blockId
// Update block — reorder agents, modify links
router.put("/loop-blocks/:blockId", async (req, res) => {
  const { blockId } = req.params;
  const { name, agents: blockAgentList, links } = req.body;

  try {
    // Update block metadata
    await db.update(operationLoopBlocks)
      .set({ name, updatedAt: new Date() })
      .where(eq(operationLoopBlocks.id, blockId));

    // Replace agents if provided
    if (blockAgentList) {
      await db.delete(loopBlockAgents).where(eq(loopBlockAgents.blockId, blockId));
      if (blockAgentList.length) {
        await db.insert(loopBlockAgents).values(
          blockAgentList.map((a: any) => ({ blockId, agentId: a.agentId, position: a.position }))
        );
      }
    }

    // Replace links if provided
    if (links) {
      await db.delete(loopBlockLinks).where(eq(loopBlockLinks.blockId, blockId));
      if (links.length) {
        await db.insert(loopBlockLinks).values(
          links.map((l: any) => ({
            blockId,

      .where(eq(loopBlockLinks.blockId, blockId));

      .from(loopBlockAgents)
      .where(eq(loopBlockAgents.blockId, blockId))

    // Import agentLoopService for individual link execution
    const { agentLoopService } = await import("../../services/agent-tool-connector");

        link.toAgentId,
        targetId,
        currentInput,
        link.maxIterations,
        link.exitCondition

      if (loop && loop.iterations.length > 0) {
        currentInput = loop.iterations[loop.iterations.length - 1].output;
      }
    }

    await logAudit(user.id, "start_loop_block_chain", "/loop-blocks", blockId, true, req);
```

### Auto-Display Across Linked Agents

When any agent in the chain is selected on the Agents page, the `LoopBlockDiagram` highlights the entire connected chain. The diagram component uses a BFS traversal client-side:

```typescript
/**
 * Given an agent ID, find all transitively connected agents via loop partnerships.
 * Uses BFS over the loops[] config arrays.
 */
function resolveLoopChain(startAgentId: string, allAgents: any[]): string[] {
  const visited = new Set<string>();
  const queue = [startAgentId];

  while (queue.length > 0) {
    const current = queue.shift()!;
    if (visited.has(current)) continue;
    visited.add(current);

    const agent = allAgents.find(a => a.id === current);
    const config = agent?.config as AgentConfig;
    const loops = config?.loops || [];

    // Legacy fallback
    if (!loops.length && config?.loopPartnerId) {
      loops.push({
        partnerId: config.loopPartnerId,
        maxIterations: config.maxLoopIterations || 5,
        exitCondition: config.loopExitCondition || "functional_poc",
      });
    }

    for (const loop of loops) {
      if (!visited.has(loop.partnerId)) {
        queue.push(loop.partnerId);
      }
    }

    // Also check agents that reference this agent as a partner
    for (const other of allAgents) {
      const otherConfig = other.config as AgentConfig;
      const otherLoops = otherConfig?.loops || [];
      if (otherConfig?.loopPartnerId === current && !visited.has(other.id)) {
        queue.push(other.id);
      }
      for (const ol of otherLoops) {
        if (ol.partnerId === current && !visited.has(other.id)) {
          queue.push(other.id);
        }
      }
    }
  }

  return Array.from(visited);
}
```

This utility is used by both the `LoopBlockDiagram` component (for highlighting) and the agent card rendering (to show the LOOP dot indicators):

- **LOOP ●** = agent participates in 1 loop chain
- **LOOP ●●** = hub agent, participates in 2+ loop partnerships

## Integration with Phase 1, 2, and 3

### Phase 1 (Mem0 Memory System)

- Floating chat can display memory-enhanced context when answering questions
- Operator responses stored as memory entries (`memoryType: 'fact'`)
- No hard dependency — deferred until Phase 1 implementation

### Phase 2 (Agent System Architecture)

- Floating chat consumes `AgentMessage` protocol messages when Phase 2 message bus is active
- Reporter agents' reports rendered in chat feed
- Operations Manager synthesis summaries displayed in chat

### Phase 3 (Operations Management Automation)

- Floating chat displays `operations.pipelineStatus` as a progress indicator
- Pipeline events (`bbot_scan_completed`, `targets_auto_created`, etc.) appear as chat messages
- Quick actions in chat trigger Phase 3 pipeline steps (manual trigger, pause, resume)
- Auto-created targets count shown in chat notifications

### v2.4 Boundary

Phase 4 does NOT implement:
- SysReptor report embedding (v2.4 - Technical Writer Agent)
- ATT&CK Planner or Workbench embedding (separate effort)
- Flow Builder embedding (separate effort)
- ATLAS/OWASP/NIST framework pages (Phase 5)
- 3D Network Topology (Phase 7)
