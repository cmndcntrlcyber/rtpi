- Service name extraction

✅ **Workflow Logic:**
```
BBOT Scan Completes
  ↓
Find pending Medium+ vulnerabilities
  ↓
For each vulnerability:
  ├─→ Check for Nuclei template by CVE ID
  │   ├─→ No template → Delegate to R&D Team (full_pipeline)
  │   └─→ Has template → Skip R&D
  ├─→ Extract target URL
  └─→ Delegate to BurpSuite Orchestrator (investigate/validate)

```

## Architecture Decisions

### ✅ Implemented Patterns

1. **Singleton Pattern:** `burp_setup` table enforces single-row constraint
2. **State Machine:** dormant → activating → active (with error state)
3. **Health Monitoring:** Continuous MCP server health checks
4. **File-based Activation:** Uses flag file (`/opt/burp-setup/.activate`)
5. **Dormant Mode:** Container runs monitoring loop until activated
6. **Admin-only Access:** All upload/activation endpoints require admin role
7. **Audit Trail:** All operations logged to `audit_logs` table
8. **Graceful Degradation:** Agents report blockers if Burp is dormant

### Infrastructure Benefits

- **No hard-coded JAR:** Users provide their own BurpSuite Pro license
- **Zero attack surface when dormant:** Burp not running until needed
- **Activation workflow prevents unauthorized use:** Admin-only controls
- **Health monitoring:** Automatic detection of MCP failures
- **Clean deactivation:** Proper shutdown and state reset

## Known Limitations

1. **MCP Server Dependency:** Requires BurpSuite MCP Server BApp (not yet installed in container)
   - **Solution:** Add MCP Server installation to Dockerfile or entrypoint script

2. **File Persistence:** Uploaded files stored in `/opt/burp-setup` volume
   - **Consideration:** May need cleanup strategy for old JARs

3. **Activation Timeout:** Fixed 2-minute timeout for MCP health check
   - **Consideration:** May need adjustment for slower systems

4. **Single Burp Instance:** One BurpSuite instance per RTPI deployment
   - **Future Enhancement:** Multi-instance support for parallel scanning
