## Overview

The Operations Manager is the central coordination hub for the Red Team Portable Infrastructure (RTPI). It orchestrates autonomous agent workflows, synthesizes multi-source intelligence, manages human-in-the-loop interactions, and delegates tasks across the system.

### Core Components

```
┌─────────────────────────────────────────────────────────────┐
│                    OPERATIONS MANAGER                       │
│                  (Central Coordination Hub)                 │
├─────────────────────────────────────────────────────────────┤
│  • Synthesis Engine (AI-powered report aggregation)         │
│  • Task Delegation System (message bus coordination)        │
│  • Question Generation (human-in-the-loop)                  │
│  • Loop Monitoring (non-productive loop termination)        │
│  • Memory Integration (context persistence)                 │
│  • Interactive Chat (OpenAI/Anthropic inference)            │
└─────────────────────────────────────────────────────────────┘
           │
           ├─────────────────┬─────────────────┬─────────────
           ▼                 ▼                 ▼
    ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
    │   Page       │  │ Vulnerability│  │   Agent      │
    │  Reporters   │  │   Reporter   │  │  Message Bus │
    │  (8 agents)  │  │ (Specialized)│  │ (Broadcast)  │
    └──────────────┘  └──────────────┘  └──────────────┘
```

### Key Statistics

- **9 Total Reporter Agents** (8 page + 1 vulnerability)
- **1 Operations Manager Agent**
- **6 Communication Channels**
- **4 Primary Workflows**
- **9 Data Sources** for chat interface

## Current Interactions

### 1. Page Reporter Agents (8 Agents)

Each reporter monitors a specific system page and generates hourly activity reports:

| Reporter | Page Role | Responsibilities | Output Frequency |
|----------|-----------|------------------|------------------|
| **Dashboard Reporter** | `dashboard` | System-wide metrics, health status | Hourly |
| **Operations Reporter** | `operations` | Operation status, progress tracking | Hourly |
| **Targets Reporter** | `targets` | Target inventory, engagement status | Hourly |
| **Vulnerabilities Reporter** | `vulnerabilities` | Vulnerability counts, severity distribution | Hourly |
| **Assets Reporter** | `surface_assessment` | Asset discovery, service enumeration | Hourly |
| **Tools Reporter** | `tools` | Tool availability, usage statistics | Hourly |
| **Workflows Reporter** | `workflows` | Workflow execution, completion status | Hourly |
| **Agents Reporter** | `agents` | Agent health, task completion | Hourly |

**Database Table:** `agents` (with `config.role = 'page_reporter'`)

**Implementation:**
- Service: `server/services/page-reporter-agent.ts`
- Individual Reporters: `server/services/reporters/*.ts`
- Base Class: `server/services/reporters/base-reporter.ts`

### 2. Vulnerability Reporter Agent (Specialized)

**Purpose:** Real-time vulnerability monitoring and alerting

**Polling Behavior:**
- Polls every 60 seconds (1 minute)
- Detects new critical/high severity vulnerabilities
- Monitors remediation status changes
- Submits urgent questions to Operations Manager

**Database Tables:**
- `agents` (vulnerability reporter agent)
- `reporters` (reporter record for tracking)

**Implementation:** `server/services/vulnerability-reporter-agent.ts`

### 3. Operations Manager Agent

**Purpose:** Central coordination, synthesis, and decision-making

**Responsibilities:**
- Synthesize hourly reports from all 8 page reporters
- Generate questions when ambiguity is detected
- Delegate tasks to appropriate agents
- Monitor agent loops for productivity
- Maintain memory context across operations
- Provide interactive chat interface

**Database Tables:**
- `agents` (Operations Manager agent)
- `operationsManagerTasks` (synthesis & coordination tasks)
- `assetQuestions` (questions for human operators)
- `memoryEntries` (contextual knowledge storage)

**Implementation:** `server/services/operations-manager-agent.ts`

## Communication Channels

### Existing Channels

```
Flow: Reporter Agent → Database → Operations Manager

```

**Purpose:** Hourly status updates from each page reporter

**Data Structure:**
```typescript
{
  agentId: uuid,
  operationId: uuid,
  agentPageRole: string,
  reportType: "hourly_status",
  activitySummary: string,
  keyMetrics: json,

  synthesizedByManagerTaskId: uuid | null

```

**Orchestration:** `server/services/ops-management-orchestrator.ts`

#### 2. **Asset Questions (Manager → Human)**
```
Flow: Operations Manager → Database → Human Operator → Manager
Table: assetQuestions
```

**Purpose:** Human-in-the-loop decision points

**Question Types:**
- `context` - Clarify scope or boundaries
- `scope` - Define what's in/out of scope
- `priority` - Determine task prioritization
- `classification` - Categorize findings
- `verification` - Confirm automated decisions

**Data Structure:**
```typescript
{
  assetId: uuid | null,
  operationId: uuid,
  question: string,
  questionType: QuestionType,
  askedBy: uuid, // Manager agent ID

  answerSource: "user" | "agent" | "automated",
  answeredByUserId: uuid | null,
  status: "pending" | "answered" | "dismissed" | "escalated",
  relevantMemoryIds: array,
  answerStoredAsMemoryId: uuid | null
```

```
Flow: Reporter Agent → Database → Human Operator

```

**Purpose:** Reporter-specific clarifications

**Data Structure:**
```typescript
{
  reporterId: uuid,
  operationId: uuid,
  question: string,
  context: json,
  priority: integer (1-10),
  status: "pending" | "answered" | "dismissed" | "escalated",

```

#### 4. **Reporter Tasks (Manager → Reporter)**
```
Flow: Operations Manager → Database → Reporter Agent
Table: reporterTasks
```

**Purpose:** Task delegation from manager to reporters

**Task Types:**
- `poll` - Collect latest data
- `analyze` - Deep analysis request
- `report` - Generate specific report
- `investigate` - Investigate anomaly
- Custom types as needed

**Data Structure:**
```typescript
{
  reporterId: uuid,
  questionId: uuid | null,
  operationId: uuid,
  taskName: string,
  taskDescription: string,

  priority: integer (1-10),

```

#### 5. **Agent Message Bus (Agent ↔ Agent)**
```
Flow: Any Agent → Message Bus → Subscribed Agents
Table: agentMessages
Service: server/services/agent-message-bus.ts
```

**Purpose:** Inter-agent communication for coordination

**Message Types:**
- `report` - Status/progress reports
- `task` - Task assignments
- `question` - Cross-agent questions
- `response` - Replies to questions
- `alert` - Urgent notifications
- `status` - Health/availability updates
- `data` - Shared data transmission

**Routing Mechanisms:**
- Direct (toAgentId specified)
- Role-based broadcast (broadcastToRole specified)
- Subscription-based filtering

#### 6. **Memory Service (Persistent Context)**
```
Flow: Any Component → Memory Service → Database
Tables: memoryContexts, memoryEntries, memoryRelationships
Service: server/services/memory-service.ts
```

**Purpose:** Long-term contextual knowledge storage

**Context Types:**
- `operation` - Operation-specific knowledge
- `target` - Target-specific intel
- `agent` - Agent-learned patterns
- `user` - User preferences
- `workflow` - Workflow patterns
- `global` - System-wide knowledge

**Memory Types:**
- `fact` - Established facts
- `event` - Historical events
- `insight` - Derived insights
- `pattern` - Identified patterns
- `procedure` - Learned procedures
- `preference` - User/system preferences

### Communication Channels Needed

#### 1. **Real-Time Status Updates**
**Current Gap:** No real-time status push mechanism  
**Proposed:** WebSocket channel for live updates  
**Use Case:** Dashboard updates, workflow progress, agent status changes

#### 2. **Priority Escalation Pathway**
**Current Gap:** Questions can be marked urgent, but no notification system  
**Proposed:** Notification service integration

#### 3. **Cross-Operation Coordination**
**Current Gap:** Operations are isolated

#### 4. **Structured Action Feedback**
**Current Gap:** No confirmation that delegated tasks were executed  
**Proposed:** Task execution acknowledgment protocol  
**Use Case:** Closed-loop task management, workflow reliability

## Workflows & Orchestration

### Workflow 1: Hourly Operations Management Cycle

**Trigger:** Hourly cron job (ops-manager-scheduler)  
**Duration:** ~5-10 minutes  
**Concurrency:** High (8 parallel reporters + 1 sequential manager)

```
Step 1: Scheduler Trigger
   └─> Check: Operation.hourlyReportingEnabled === true
       └─> Filter: No active "ops_management_hourly" workflow running

Step 2: Workflow Creation
   └─> Create agentWorkflow record
   └─> Create 9 workflowTasks:
       ├─> 8 Reporter Tasks (sequence 1-8, parallel)
       └─> 1 Manager Task (sequence 9, sequential after reporters)

Step 3: Reporter Execution (Parallel)
   └─> For each of 8 reporters:
       ├─> Execute reporter.executeReport(operationId, reportingPeriod)
       ├─> Collect page-specific data
       ├─> Detect changes from previous poll
       ├─> Identify issues & recommendations
       ├─> Insert agentActivityReport record
       └─> Update task status to "completed"

Step 4: Manager Synthesis (Sequential)
   └─> Triggered after ALL reporters complete
   └─> Steps:
       ├─> Fetch all 8 activity reports
       ├─> Query memory for operation context
       ├─> AI Synthesis (OpenAI/Anthropic/Ollama):
       │   └─> Prompt: Combine 8 reports into summary + insights
       ├─> Create operationsManagerTasks record (type: "synthesis")
       ├─> Store synthesis in memory
       ├─> Update Operation.lastSynthesisAt
       └─> Mark all reports as "synthesized"

Step 5: Workflow Completion
   └─> Update agentWorkflow.status = "completed"
   └─> Set progress = 100
```

**Implementation:**
- Scheduler: `server/services/ops-manager-scheduler.ts`
- Orchestrator: `server/services/ops-management-orchestrator.ts`
- Manager: `server/services/operations-manager-agent.ts`

### Workflow 2: Question-Answer-Action Cycle

**Trigger:** Reporter detects ambiguity OR Manager generates question  
**Duration:** Variable (waits for human response)  
**Interaction:** Human-in-the-loop

```
Step 1: Question Generation
   └─> Trigger Source:
       ├─> Reporter detects unusual pattern
       ├─> Manager detects ambiguity in synthesis
       ├─> Vulnerability Reporter finds critical finding
       └─> Explicit API call

Step 2: Question Submission
   └─> Insert assetQuestions OR reporterQuestions
   └─> Set priority (1-10, higher = more urgent)
   └─> Link to relevant memories (relevantMemoryIds)

Step 3: Human Response
   └─> Operator provides answer via UI
   └─> Answer stored in question record
   └─> Status changed to "answered"

Step 4: Manager Processing
   └─> operationsManagerAgent.processResponse()
   ├─> Store answer in memory (memoryType: "fact")
   ├─> Link memory to question (answerStoredAsMemoryId)
   └─> Optional: Generate follow-up task

Step 5: Task Generation (Optional)
   └─> If answer indicates action needed:
       ├─> Create reporterTask
       ├─> Assign to appropriate reporter
       └─> Link task to originating question

Step 6: Task Execution
   └─> Reporter polls for assigned tasks
   └─> Executes task according to instructions
   └─> Reports completion/failure
```

**Implementation:**
- Question Gen: `operations-manager-agent.ts::generateQuestion()`
- Response Processing: `operations-manager-agent.ts::processResponse()`
- Task Assignment: `reporter-agent-service.ts::assignTask()`

### Workflow 3: Agent Loop Monitoring

**Trigger:** 30-second interval check  
**Duration:** Continuous  
**Purpose:** Prevent non-productive infinite loops

```
Step 1: Periodic Check
   └─> Every 30 seconds, evaluateActiveLoops()

Step 2: Loop Analysis
   └─> For each running loop:
       ├─> Check iteration progress vs max
       ├─> Detect repeating output patterns
       ├─> Calculate progress rate (iterations/minute)
       └─> Assess elapsed time vs meaningful progress

Step 3: Termination Decision
   └─> Conditions for termination:
       ├─> Approaching max iterations (>80%) without exit condition
       ├─> Repeating outputs (last 3 iterations identical)
       ├─> Progress rate < 0.5 iterations/min
       └─> Running >10 min with <3 iterations

Step 4: Loop Termination
   └─> If decision.terminate === true:
       ├─> agentLoopService.stopLoop(loopId)
       ├─> Emit "loop_terminated" event
       └─> Log termination reason
```

**Implementation:** `operations-manager-agent.ts::evaluateActiveLoops()`

### Workflow 4: Task Delegation via Message Bus

**Trigger:** Manager determines agent action needed  
**Duration:** Async, depends on agent availability  
**Purpose:** Coordinate multi-agent workflows

```
Step 1: Task Creation
   └─> Create operationsManagerTasks record
   └─> Define:
       ├─> taskType: "coordination"
       ├─> targetAgentRole: "web_hacker" | "researcher" | etc.
       ├─> parameters: Task-specific data
       └─> priority: 1-10

Step 2: Message Broadcasting
   └─> agentMessageBus.sendMessage()
   ├─> messageType: "task"
   ├─> broadcastToRole: targetAgentRole
   ├─> subject: taskName
   └─> content: { taskId, parameters }

Step 3: Agent Subscription Matching
   └─> Query agentMessageSubscriptions
   └─> Match agents with:
       ├─> messageType = "task"
       ├─> fromAgentRole = "operations_manager"
       └─> isActive = true

Step 4: Task Pickup
   └─> Subscribed agents receive message
   └─> Agent processes task
   └─> Agent sends response message (optional)

Step 5: Completion Tracking
   └─> Currently: No automatic tracking
   └─> Future: Task acknowledgment protocol needed
```

**Implementation:**
- Delegation: `operations-manager-agent.ts::delegateTask()`
- Message Bus: `server/services/agent-message-bus.ts`

### Workflow 5: Interactive Chat (NEW - v2.4.0)

**Trigger:** User sends chat message  
**Duration:** 1-5 seconds (AI inference time)  
**Purpose:** Real-time Operations Manager consultation

```
Step 1: User Input
   └─> User types message in chat interface
   └─> Frontend: OpsManagerFloatingChat component
   └─> Message includes conversation history

Step 2: Context Aggregation
   └─> Backend fetches 9 data sources:
       ├─> 1. Current operation details
       ├─> 2. Global operations stats
       ├─> 3. Targets summary
       ├─> 4. Vulnerabilities breakdown
       ├─> 5. Workflows status
       ├─> 6. Reporter agents info
       ├─> 7. Recent activity reports (last 10)
       ├─> 8. Questions (pending & answered)
       └─> 9. Latest synthesis output

Step 3: AI Inference
   └─> Build system prompt with full context
   └─> Add conversation history
   └─> Call ollamaAIClient.complete():
       ├─> Auto-selects provider (OpenAI > Anthropic > Ollama)
       ├─> Model: gpt-5.2-chat-latest OR claude-sonnet-4-5
       ├─> Temperature: 0.7
       └─> Max tokens: 1024

Step 4: Response Delivery
   └─> Return AI response to frontend
   └─> Display in chat bubble
   └─> Maintain conversation history for follow-ups
```

**Implementation:**
- Frontend: `client/src/components/operations/OpsManagerFloatingChat.tsx`
- Backend: `server/api/v1/operations-management.ts::/chat`
- AI Client: `server/services/ollama-ai-client.ts`

## Action Triggers

### Status Update Actions

#### Workflow Completion
```
Trigger: agentWorkflow.status = "completed"
Actions:
  1. Check if workflow.workflowType === "ops_management_hourly"
  2. If yes:
     └─> Mark all activity reports as reviewed
     └─> Update operation.lastSynthesisAt
  3. Emit "workflow_completed" event
```

#### Workflow Failure
```
Trigger: agentWorkflow.status = "failed"
Actions:
  1. Generate assetQuestion asking for operator intervention
  2. Log failure details
  3. Optionally: Auto-retry based on failure type
```

#### Critical Vulnerability Detected
```
Trigger: vulnerability.severity = "critical" AND status = "open"
Actions (Vulnerability Reporter):
  1. Submit URGENT reporterQuestion:
     └─> Priority: 10 (highest)
     └─> Question type: "verification"
     └─> Message: "CRITICAL vulnerability detected: {title}. Immediate attention required."
  2. Emit event for notification system
  3. Optionally: Trigger immediate investigation workflow
```

#### Reporter Failure
```
Trigger: Reporter polling error OR task failure
Actions:
  1. Update reporter.status = "error"
  2. Log error message
  3. Attempt restart if autoRestart enabled
  4. If repeated failures (>3):
     └─> Generate assetQuestion for manual intervention
```

#### Loop Stagnation
```
Trigger: Operations Manager detects non-productive loop
Actions:
  1. Automatic termination
  2. Emit "loop_terminated" event
  3. Log termination reason
  4. Optionally: Generate question asking if loop should be reconfigured
```

#### Hourly Reporting Toggle
```
Trigger: operation.hourlyReportingEnabled changes
Actions:
  If enabled:
    1. Include operation in next hourly cycle
    2. Initialize reporter agents if needed
  If disabled:
    1. Exclude operation from hourly cycles
    2. Keep reporters active but skip workflow creation
```

### Completion Actions

#### Reporter Task Completion
```
Trigger: reporterTask.status = "completed"
Actions:
  1. Update reporter.lastActivity
  2. Increment reporter.totalTasks
  3. If linked to question:
     └─> Update reporterQuestion.status
  4. Store result data in memory (if significant)
```

#### Synthesis Completion
```
Trigger: operationsManagerTask (type: "synthesis") completes
Actions:
  1. Store synthesis in operation.synthesisSummary
  2. Create memory entry with insights
  3. Mark all source reports as "synthesized"
  4. Increment workflow progress to 100%
  5. Update operation.lastSynthesisAt
```

#### Question Answer Stored
```
Trigger: assetQuestion.status = "answered"
Actions:
  1. Create memory entry from Q&A pair
  2. Link memory to question
  3. If answer indicates action:
     └─> Generate reporterTask for follow-up
  4. Update analytics counters
```

#### Reporter Implementation Pattern

**Base Class:** `BaseReporter` (`server/services/reporters/base-reporter.ts`)

```typescript
abstract class BaseReporter {
  protected abstract pageRole: string;
  
  abstract collectData(operationId: string): Promise<PageData>;
  
  async executeReport(operationId: string, reportingPeriod: string): Promise<void> {
    // 1. Collect page data
    const data = await this.collectData(operationId);
    
    // 2. Compare with previous
    const changes = await this.detectChanges(operationId, data);
    
    // 3. Identify issues
    const issues = await this.identifyIssues(data);
    
    // 4. Generate recommendations
    const recommendations = await this.generateRecommendations(data, issues);
    
    // 5. Create activity summary
    const summary = await this.createSummary(data, changes, issues);
    
    // 6. Store report
    await this.storeReport(operationId, {
      activitySummary: summary,
      keyMetrics: data.metrics,
      changesDetected: changes,
      issuesReported: issues,
      recommendations: recommendations
    });
  }
}
```

**Scaling:** To add a new reporter:
1. Create class extending `BaseReporter`
2. Implement `collectData()` method
3. Define page-specific metrics
4. Create agent record with `config.role = 'page_reporter'`
5. Orchestrator automatically includes in hourly cycle

## Scaling Considerations

### Current Scale
- **8 Page Reporters** (one per major system page)
- **1 Vulnerability Reporter** (specialized polling)
- **1 Operations Manager** (synthesis & coordination)
- **Hourly cycle processes ~8-10 reports**
- **Chat handles unlimited concurrent conversations**

### Scaling Mechanisms

#### Horizontal Reporter Scaling
```
Pattern: Add reporters for new pages/domains

Steps:
1. Create new class extending BaseReporter
2. Define page-specific collectData() logic
3. Set pageRole identifier
4. Create agent record in database
5. Automatic inclusion in orchestration

Max Recommended: 20 concurrent reporters
Bottleneck: AI synthesis time increases linearly
```

#### Vertical Operations Scaling
```
Pattern: Multiple concurrent operations

Current: Each operation has independent hourly cycles
Capacity: No hard limit (database-bound)
Consideration: 
  - Synthesis is operation-specific (no cross-contamination)
  - Memory contexts are isolated per operation
  - Reporter agents are shared across operations

Recommended: <50 active operations for optimal performance
```

#### Chat Scaling
```
Pattern: Stateless per-request design

Characteristics:
  - Each chat request fetches fresh context
  - No session state stored server-side
  - Conversation history sent from client
  - AI inference auto-scales with provider limits

Capacity: Limited only by AI provider rate limits
OpenAI: ~10,000 RPM (requests per minute)
Anthropic: ~5,000 RPM
```

#### Memory System Scaling
```
Current: PostgreSQL with JSON embeddings
Future: Migrate to pgvector for efficient similarity search

Scaling Path:
  Phase 1: JSON arrays (current) - Good to ~10K memories
  Phase 2: pgvector extension - Good to ~1M memories
  Phase 3: Dedicated vector DB - Unlimited scale
```

### Performance Optimization

**Reporter Execution:**
- All 8 reporters run in **parallel** (Promise.allSettled)
- Graceful degradation: Manager proceeds if ≥1 reporter succeeds
- Timeout: 5 minutes per reporter (configurable)

**Synthesis:**
- AI provider auto-selection for redundancy
- Response caching for identical prompts (1 hour TTL)
- Fallback to simple concatenation if AI fails

**Memory Queries:**
- Limit: 20 memories per context fetch
- Indexed by contextId for fast lookup
- Relevance scoring via embeddings (future)

## Database Schema Summary

### Core Tables

| Table | Purpose | Key Fields | Relationships |
|-------|---------|------------|---------------|
| `agents` | Agent definitions | id, name, type, config | → Many to agentActivityReports |
| `reporters` | Reporter tracking | id, agentId, pageId, status | → Many to reporterQuestions, reporterTasks |
| `agentActivityReports` | Hourly status reports | agentId, operationId, activitySummary | ← agents, operations |
| `assetQuestions` | Manager-generated questions | operationId, question, answer | ← operations, agents |
| `reporterQuestions` | Reporter-generated questions | reporterId, question, response | ← reporters |
| `reporterTasks` | Tasks assigned to reporters | reporterId, taskName, status | ← reporters |
| `operationsManagerTasks` | Manager's tasks | taskType, operationId, outputData | ← operations, agents |
| `agentMessages` | Message bus records | fromAgentId, toAgentId, messageType | ← agents |
| `agentMessageSubscriptions` | Message routing | agentId, messageType, isActive | ← agents |
| `memoryEntries` | Persistent knowledge | contextId, memoryText, memoryType | ← memoryContexts |

### Data Retention

- **Activity Reports:** Retained indefinitely (for trend analysis)
- **Questions:** Retained indefinitely (for audit trail)
- **Messages:** 30-day retention (configurable)
- **Memory:** Indefinite with periodic pruning of low-relevance entries
- **Chat History:** Client-side only (not persisted server-side)

## Troubleshooting Guide

### Common Issues

**Issue:** Reporter stuck in "polling" status  
**Cause:** Reporter crashed mid-execution  
**Solution:** Restart reporter via `reporterAgentService.startPolling(reporterId)`

**Issue:** Manager synthesis returns empty summary  
**Cause:** AI provider failure without fallback  
**Solution:** Check AI provider configuration; ensure Ollama is running as fallback

**Issue:** Questions not appearing in UI  
**Cause:** Operation not selected OR status filter applied  
**Solution:** Select operation card, ensure "Features" tab is active

**Issue:** Chat not responding  
**Cause:** Missing AI API keys OR /chat endpoint not registered  
**Solution:** Verify OpenAI/Anthropic API keys in .env, check server logs
